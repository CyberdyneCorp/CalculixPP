## 3. Heat transfer (spec: heat-transfer-analysis)

- [x] 3.1 Steady-state conduction element kernels; `*HEAT TRANSFER, STEADY STATE` — scalar 1-DOF/node temperature field parallel to the mechanical path: `fem::element_conduction` (Kt_e = Σ_gp (g_a·g_b) k detJ w, reusing the shared shape functions/Gauss/`physical_gradients`), `fem::assemble_conduction` (thermal DOF numbering + Dirichlet elimination of prescribed temperatures), `*CFLUX`/`*DFLUX` into the thermal rhs (`fem::thermal_load_vector`; DFLUX reuses a generalized face integrator now covering tet/hex/wedge faces), and `numerics::solve_heat_transfer` (SPD Kt T = q via `solve_reduced`). CLI + Python auto-dispatch on `*HEAT TRANSFER`. VALIDATED against stock CalculiX `oneel20cf` (*CFLUX, NT abs err ~1.8e-15, RFL ~3.6e-14) and `oneel20df` (*DFLUX S1 hex-face, NT ~1.3e-15) plus an analytical 1-D bar (linear profile + Fourier Q=kAΔT/L). Transient (3.2)/film (3.3)/radiation (3.4) remain deferred.
- [x] 3.2 Transient conduction (backward Euler) — capacitance `C_e = ∫ rho*c N_a N_b dV` (`fem::element_capacitance`, `Model::element_heat_capacity` from `*SPECIFIC HEAT` × `*DENSITY`) assembled on the full node numbering (`fem::assemble_full_thermal`) and time-stepped by `numerics::solve_transient`: backward Euler `(C/dt + Kt_film) T_{n+1} = (C/dt) T_n + q`, marching the step period (`increment.total`) in `increment.initial` sub-steps. Parser now accepts a transient `*HEAT TRANSFER` (no STEADY STATE) and `*INITIAL CONDITIONS, TYPE=TEMPERATURE`; CLI + Python auto-dispatch. VALIDATED analytically: (a) a unit cube convecting on all six faces keeps the field spatially UNIFORM, so the FEM reduces EXACTLY to the lumped ODE `Cv dT/dt = -h A (T-T_sink)` — the driver reproduces the backward-Euler recurrence to ~1e-8 (accounting for the default linear ramp of the unamplified sink); (b) a long transient relaxes to the independently-computed steady conduction+film field (~1e-3). (tests/test_thermal_transient.cpp)
- [x] 3.3 Convective film loads (`*FILM`) — `q_film = h (T - T_sink)` on a face adds `h ∫N_a N_b dA` to `Kt` and `h T_sink ∫N_a dA` to the rhs, via the shared face surface integrator `fem::face_surface_integrals` (reuses the `*DFLUX`/pressure face quadrature, all tet/hex/wedge faces). Wired into both the steady (`assemble_conduction` / `assemble_full_thermal`) and transient paths; parser handles `*FILM elem,F<face>,sink,h`. VALIDATED against stock CalculiX `oneel20fi` (single C3D20R cube, `*FILM 1,F1,-12,10`): nodal temperatures (NT −2/−1/0 profile) and heat generation (RFL, conduction-only) match to ~1e-14 (Python regression corpus). Forced-convection NETWORK film coupling is out of scope (Phase-5 1-D networks) — solid-only surface film done.
- [~] 3.4 Radiation (`*RADIATE`) — surface-to-ambient `q = eps sigma (T^4 - T_amb^4)` implemented and linearized (Newton on `dq/dT = eps sigma 4 T_abs^3`) inside the thermal iteration, in absolute temperature via `*PHYSICAL CONSTANTS` (ABSOLUTE ZERO / STEFAN BOLTZMANN); works in both steady and transient. Parser handles `*RADIATE elem,R<face>,ambient,eps`. VALIDATED analytically against a conduction↔radiation equilibrium (`k(T_hot−T)/L = eps sigma (T_abs^4−T_amb_abs^4)`, scalar-root reference, ~1e-3). CAVITY radiation (`*RADIATE ...,CR`, O(n²) view factors + gray-body exchange) is NOT implemented — the card is rejected with an actionable message; deferred to a dedicated view-factor task. (spec: heat-transfer-analysis — radiation with view factors.)

## 4. Coupled thermomechanics (spec: heat-transfer-analysis — coupled)

- [~] 4.1 `*COUPLED TEMPERATURE-DISPLACEMENT` — ONE-WAY (sequential) coupling implemented end to end and validated; monolithic tangent deferred. The parser recognizes `*COUPLED TEMPERATURE-DISPLACEMENT` (`Procedure::Coupled`), routing the thermal cards (temperature BCs on DOF 11, `*CFLUX`/`*DFLUX`/`*FILM`) to the scalar field and the mechanical loads to the displacement field. `numerics::solve_coupled` solves the steady thermal field (reusing `solve_heat_transfer`), copies the nodal temperatures into `Model::applied_temperature`, then runs the mechanical `solve_linear_static` so the thermal strain eps_th = alpha (T - Tref) induces the thermal stress; it returns a `CoupledFields{thermal, mechanical}`. CLI + Python (`solve`/`solve_text`) auto-dispatch on the procedure and the result dict carries both temperature and displacement/stress/reaction. VALIDATED analytically end to end: a bar with a prescribed linear temperature profile and u_x fixed on both end faces develops sigma_xx = -E alpha T_bar (tests/test_thermal_stress.cpp test_coupled_deck_end_to_end + python test_coupled_temperature_displacement_thermal_stress). A full MONOLITHIC 4-DOF/node tangent (temperature<->displacement in one Newton system, needed when the mechanical work feeds back into the thermal field) is deferred — the one-way scheme captures the dominant thermal->mechanical coupling for thermal-stress problems.
- [~] 4.2 Staggered coupling fallback — NOT separately implemented. The one-way (thermal-then-mechanical) scheme in 4.1 IS a degenerate single-pass staggered scheme; a true iterated (Gauss-Seidel) staggered loop with a two-way convergence check is deferred together with the monolithic tangent (both require the mechanical->thermal feedback path).
- [x] 4.3 Thermal expansion coupling verified against reference thermal-stress decks — `*EXPANSION[, ZERO=Tref]` parsed into `Material::expansion` (isotropic alpha + reference/stress-free temperature Tref) and consumed by the mechanical path via `Model::element_expansion()`. The equivalent thermal-strain load f_th = ∫ Bᵀ D (alpha (T - Tref)) dV (`fem::element_thermal_load` / `fem::thermal_strain_load_vector`, reusing the shared shape functions / Gauss / physical_gradients) is added to the mechanical rhs, and stress recovery uses sigma = D (eps_mech - eps_th), so a temperature field induces the correct thermal stress/strain while the internal force f_int = Ke u - f_th keeps the reactions RF consistent. Gated on `Model::has_thermal_strain()` so a deck with no applied temperature is byte-for-byte the pure-mechanical path (all Phase-1/2 reference matches unchanged). VALIDATED (a) ANALYTICALLY: a fully axially-constrained bar heated by dT develops sigma_xx = -E alpha dT (and sigma_yy=sigma_zz=0), and a free bar expands by alpha dT L with zero stress (tests/test_thermal_stress.cpp); (b) against a REFERENCE DECK: stock CalculiX `beamt` (C3D20R "heated beam between two fixed walls", `*EXPANSION,ZERO=293` + `*TEMPERATURE 500`) reproduces the `.dat.ref` nodal displacements to rel-L2 ~2.4e-6 (python REFERENCE_CORPUS). Temperature-dependent alpha(T) tables are not yet consumed (only the first isotropic coefficient) — same limitation noted for conductivity/specific-heat in 6.1.

## 5. Model change (spec: model-change)

- [~] 5.1 `*MODEL CHANGE, TYPE=ELEMENT` deactivate / reactivate element sets (strain-free reactivation) — WITHIN-STEP birth-death implemented and validated; TRUE cross-step birth-death deferred to the multi-step engine. A per-element active mask (`Model::deactivated_elements` / `Model::element_active_mask()`, aligned with `mesh.elements()`) is consulted by every assembled system: `REMOVE`d elements contribute no stiffness/internal force (`assemble_linear_static`, `assemble_material_tangent`), no stress/reaction (`stress.cpp` recover_fields + internal_force), no body/pressure/thermal-strain load (`loads.cpp`), and no conduction/capacitance (`thermal.cpp` — the thermal path is gated too for uniformity). `TYPE=ELEMENT, ADD` for an id drops it from the deactivated set (reactivation). The parser accepts `*MODEL CHANGE, TYPE=ELEMENT, ADD|REMOVE` with element ids or elset names on the data lines. The mask is all-true when no element is removed, so a deck without `*MODEL CHANGE` is byte-for-byte the pre-model-change mechanical path (all Phase-1/2 reference matches + the nonlinear-reproduces-linear gate unchanged). VALIDATED ANALYTICALLY (tests/test_model_change.cpp + python test_model_change_element_birth_death_python): two confined unit C3D8 cubes (each a unit-area uniaxial specimen, fixed-face reaction −M·delta with M the confined modulus) — REMOVE one cube HALVES the fixed-face x-reaction (removed element carries no load: "creates a hole"), and REMOVE-then-ADD reproduces the never-removed model IDENTICALLY (reaction to ~1e-9, displacements to ~1e-12), i.e. strain-free reactivation. Strain-free is exact here because the single-step model always solves from the undeformed state. TRUE cross-step birth-death — an element active in step 1, removed in step 2, then re-added strain-free relative to the DEFORMED geometry at reactivation — requires the multi-step step-history engine (openspec/BACKLOG.md, same enabler as OP=MOD/NEW and *CHANGE MATERIAL cross-step); the within-step semantics above are the feasible slice.
- [~] 5.2 `*MODEL CHANGE, TYPE=CONTACT PAIR` activate / deactivate contact pairs between steps — PARSE + STORE only (contact assembly is the separate contact workflow). The parser accepts `*MODEL CHANGE, TYPE=CONTACT PAIR, ADD|REMOVE` with a `surface_a, surface_b` data line and records each as a `ContactPairChange{surface_a, surface_b, add}` on `Model::contact_pair_changes`. No contact search/force/constraint is assembled from it yet — the record is stored for the contact workflow (task 2.x) that will consult it. VALIDATED: tests/test_model_change.cpp test_contact_pair_stored (record content + ADD/REMOVE sense) and python test_model_change_contact_pair_parses_python (the deck solves through Python, the pair does not perturb the mechanical solve). Between-step activation semantics need the multi-step engine (deferred with 5.1).
